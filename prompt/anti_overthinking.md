# Anti-Overthinking Addendum

Use this addendum for low-thinking or none-thinking production runs when the model tends to overbuild the program.

Addendum:

- Use the shortest valid plan.
- Commit early to the first minimal correct structure.
- Do not explore multiple program styles.
- Do not introduce `namespace` unless collisions are unavoidable.
- Do not introduce `record` if `pack` already matches the requested output.
- Do not introduce `shape` unless nested key drift is a real risk.
- If two solutions seem valid, choose the shorter one.
- Prefer one root `fork` over a deeper tree when possible.
